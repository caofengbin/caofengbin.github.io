ButterKnife的使用与原理(2)

1.前言

  本文是上一篇文章的续篇。上次大致介绍了ButterKnife的基本使用以及通过反射技术模拟实现了简易的ButterKnife效果。但是反射多少会有性能上的损耗。ButterKnife最新版本也不是通过反射来完成的，而是利用APT(Annotation Processing Tool)编译时解析技术，动态生成绑定事件或者View的java代码。网上关于ButterKnife源码分析的文章也很多。本文试图利用apt技术自己模拟实现一个简易的ButterKnife，来提升对ButterKnife，Dagger等类似框架的认识，并详细介绍注解处理器(Annotation Processor)的使用。

2.编译时解析注解技术简介

(1) Java编译器对Annotation的处理

OpenJDK对java编译过程的解释

Java编译过程图解：



Java编译主要分为三个过程：

  • (1) Parse and Enter
    解析源文件, 生成抽象语法树(AST),就像DOM树一样, 编译器会在这个阶段检查语法错误；
  • (2) Annotation Processing
    调用自定义的AnnotationProcessor,生成新文件,包括新的源代码,如果在这个阶段有生成新的代码文件, 则会重回第一步, 直到没有新的代码文件生成为止；
  • (3) Analyse and Generate
    根据AST生成.class文件,在这一步会检查语法错误, 例如调用了没有定义的方法, 出现了不可到达的语句等, 编译器会在这一部分中做很多优化来提升运行效率(例如将for循环改成使用iterator的循环)；

  注解处理器(AnnotationProcessor)是javac的一个工具，用来在编译时扫描和处理注解。它允许我们在Java编译器编译代码之前，执行一段我们的代码。这些生成的Java代码是在生成的.java文件中，所以不能用来修改已经存在的Java类–例如向已有的类中添加方法。这些生成的Java文件，会同其他普通的手动编写的Java源代码一样被javac编译。

(2) apt(Annotation Processor Tool)相关技术简介

  从注解类型被引入的Java SE5.0到java SE7.0,自定义注解类型的处理方式发生了很大的变化：

  • (1) Java 5使用的是apt工具和Oracle私有的Mirror API来处理；
  • (2) Java 6引入了可插拔式注解处理机制，以及相应的javax,annotation.processing和javax.lang.model包。
  • (3) Java 7把java 5中的apt工具和对应的Mirror API声明为废弃，不推荐使用；

  关于编译时处理Annotation，主要问题在于自己如何实现apt(Annotation Processor Tool)的processor，以及编译环境如何使用它。一个通用的流程如下所示:

  • (1) 自定义Annotation；
  • (2) 通过apt提供的接口，自定义Annotation processor类，实现主要的处理方法；
  • (3) 注册自定义的Processor，以便apt识别；
  • (4) 在使用的module内指定使用apt来编译该processor。

下面我们就会按照该流程利用apt技术简单的模拟ButterKnife的效果。

(2) Elements的作用

在init()方法中，我们通过

  得到一个Elements的引用。它是后续处理的关键。
  在注解处理过程中，我们扫描所有的Java源文件。源代码的每一个部分都是一个特定类型的Element。对于编译器来说，代码中的元素结构是基本不变的，例如，组成代码的基本元素有包，类，函数，字段，字段参数，变量。JDK中为这些元素定义了一个基类，也就是Element类，它有如下几个子类：

  • (1) PackageElement——包元素，包含了某个包下的信息，可以获取到包名等；
  • (2) TypeElement——类型元素，如某个字段属于某种类型；
  • (3) ExecutableElement——可执行元素，代表了函数类型的元素；
  • (4) VariableElement——变量元素；
  • (5) TypeParameterElement——类型参数元素。

6.代码生成

(1) 生成代码的方法

  现在所有注解的类信息均保存为相应的数据模型，并且组合到了ButterKnifeAnnotatedClass中，接下来就需要生成生成Java文件了，生成java代码的方法目前主要有如下的几种方式：

  • 使用Filer提供的Writer对象，通过连接字符串来生成的Java代码(这个Filer就是在init()方法中获取到的),实际操作起来比较繁琐；
  • 使用JavaWriter生成java源文件；
  • 使用codemodel生成Java源文件;
  • 使用Square公司javapoet生成Java源文件;

关于后三个开源项目的使用对比，有兴趣的同学可以参加链接–生成Java源文件 (javawriter, javapoet, codemodel)。在本文中，我们采用通过javapoet的技术进行相应的实现。

(2) javapoet的使用

  javapoet是大名鼎鼎的square公司开源的一个项目，javapoet要比JavaWriter稍微复杂，此库定义了一系列的数据结构来表示Java源文件中某些固定的概念，如class，interface，annoation，field，method等。javapoet对java源文件的结构进行了建模，常用的几个包括：

  • MethodSpec：代表一个构造函数或方法声明。
  • TypeSpec：代表一个类，接口，或者枚举声明。
  • FieldSpec：代表一个成员变量，一个字段声明。
  • JavaFile：包含一个顶级类的Java文件。
